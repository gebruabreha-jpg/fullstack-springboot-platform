In a Spring Boot microservices setup (User, Order, Notification services + a service mesh), a key rule is: business logic must stay inside the service boundary and not leak into infrastructure or communication layers.

Here’s what you should NOT put into your Java business logic (service layer/domain layer):

1. ❌ Service mesh / infrastructure concerns

Don’t mix mesh responsibilities into business code.

Avoid in Java code:

Retry logic (handled by service mesh like Istio/Linkerd)
Circuit breakers (should be mesh or gateway, not business logic)
Load balancing logic
Traffic routing / canary logic
Service discovery calls (e.g., manually finding service instances)

👉 These belong to the service mesh or API gateway, not your Java services.

2. ❌ Inter-service communication plumbing

Don’t hardcode communication mechanics inside business logic.

Avoid:

Direct REST client configuration scattered in services
Manual retry loops for HTTP calls
Kafka/RabbitMQ connection setup inside service classes
Serialization/deserialization logic mixed with business rules

👉 Instead use:

Declarative clients (Feign/WebClient abstraction)
Messaging abstraction layers
3. ❌ Business logic from other microservices

Each service must own its domain.

Example:

Order Service should NOT calculate user discount rules (User Service responsibility)
Notification Service should NOT decide order validation rules
User Service should NOT manage order state

👉 Avoid:

Cross-service domain rules duplication
Calling another service just to “validate core logic”

Use:

API calls only for data, not for foreign business rules
4. ❌ Orchestration logic inside services (too much coupling)

Don’t turn a service into a “mini monolith orchestrator”.

Avoid:

Order Service calling User + Notification + Payment + Inventory in a tightly coupled chain with business logic decisions everywhere

Better:

Use event-driven architecture
OrderCreated → Notification Service reacts
UserUpdated → downstream consumers react
5. ❌ Transport-layer logic inside domain logic

Avoid mixing HTTP / messaging details with business rules.

Don’t do:

HTTP status code handling inside domain service logic
JSON parsing in service classes
Kafka topic names in business methods
6. ❌ Retry, timeout, fallback logic in business code

These are resilience concerns, not business rules.

Instead use:

Service mesh policies
Spring Cloud Gateway
Resilience4j (only if not using mesh)
7. ❌ Logging/metrics tied to business decisions

Avoid:

“If user is premium then log differently” (logging is cross-cutting)
Embedding audit logic inside core business methods

Use:

AOP / interceptors
Observability tools (Prometheus, OpenTelemetry)
🧠 Clean separation idea (important)
Business logic should only contain:
Domain rules (e.g., order is valid, user is active)
State transitions (Order → CREATED → PAID → SHIPPED)
Core computations (pricing, validation)
Everything else belongs to:
Service mesh (traffic, retries, routing)
API Gateway (auth routing, rate limits)
Messaging system (async communication)
Infrastructure layer (DB, HTTP, Kafka config)
