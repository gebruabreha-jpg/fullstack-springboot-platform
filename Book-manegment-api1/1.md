FastAPI Development Workflow
1. Understand the domain

Ask yourself:

What objects does my application manage?
What information do I need to store?
How are those objects related?

Example:

Book
- id
- title
- author
- isbn
- available

User
- id
- name
- email


schemas.py
Pydantic schemas define:

what the client sends
what the API returns
validation rules

Example:

    title: str
    author: str
    isbn: str


class Book(BaseModel):
    id: int
    title: str
    author: str

    class Config:
        from_attributes = True

3. Design the database

models.py

SQLAlchemy models define how data is stored.

class Book(Base):
    __tablename__ = "books"

    id = Column(Integer, primary_key=True)
    title = Column(String)
    author = Column(String)
    isbn = Column(String)

Think about:

primary keys

relationships
constraints
4. Configure the database

Create:

database.py

5. Repository layer (data access)

Repositories contain database queries only.

Example:

class BookRepository:
    create()
    get_by_id()
    get_all()
    update()
    delete()

The repository should not contain business rules.

Its job is simply:

"Talk to the database."

6. Service (business logic)

Services contain business rules.

Example:

class BookService:
    borrow_book()
    return_book()

Example logic:

if not available:
    raise BookAlreadyBorrowed()

create loan

mark book unavailable

The service coordinates repositories and applies business rules.

7. API (routers)

Routers expose HTTP endpoints.

POST /books

GET /books

GET /books/{id}

PUT /books/{id}

DELETE /books/{id}

Routers should be thin:

One suggestion

If you're following a layered architecture, I'd slightly change your final line:

Understand the domain
        ↓
Design the API contract (Pydantic schemas)
        ↓
Design the database models
        ↓
Configure the database
        ↓
Repository (data access)
        ↓
Service (business logic)
        ↓
API (routers/endpoints)
        ↓
UI

This order separates responsibilities cleanly and makes the application easier to test and maintain. It's a workflow you'll see in many production FastAPI projects.

Repository Interface (ABC)
What it means
Currently your BookService depends on the concrete BookRepository class:

# Current - tight coupling
class BookService:
    def __init__(self, repo: BookRepository):  # ← concrete class
        self.repo = repo
An interface (Abstract Base Class) defines a contract without implementation:

# interfaces/book.py
from abc import ABC, abstractmethod
from app.models.book import BookModel
from app.schemas.book import BookCreate, BookUpdate

class BookRepositoryInterface(ABC):
    @abstractmethod
    def fetch(self) -> list[BookModel]: ...
    
    @abstractmethod
    def fetch_by_id(self, id: int) -> BookModel | None: ...
    
    @abstractmethod
    def add(self, book_data: BookCreate) -> BookModel: ...
    
    @abstractmethod
    def update(self, id: int, book_data: BookUpdate) -> BookModel | None: ...
    
    @abstractmethod
    def delete(self, id: int) -> BookModel | None: ...
Then your service depends on the abstraction:

# services/book.py
class BookService:
    def __init__(self, repo: BookRepositoryInterface):  # ← depends on interface
        self.repo = repo
And you implement it:

# repositories/book.py
class BookRepository(BookRepositoryInterface):  # ← implements interface
